# 協調共有プロトコル(検討ノート)

> **状態: [構想] — これは決定でなく検討素材です。** 後から読んで腰を据えて検討するために、論点・トレードオフ・未決を正直に並べたノートです。実装を拘束するのは [DESIGN.md](DESIGN.md) と [ROADMAP.md](../ROADMAP.md) のみで、本書の内容は採用が決まった部分だけそちらへ昇格させます。最終更新: 2026-06-25。

## 0. 一行で

**中央にデータを集めるのではなく、形式(スキーマ)を標準化して各自が追記で持ち寄る。** データの重力を中央に作らない。

## 1. 何を共有するのか — 2層に分かれる

| | ノード共有 | **エッジ共有** |
|---|---|---|
| 中身 | 採寸結果(素体/衣装の周径スカラー＋メッシュハッシュ＋計測条件) | **フィット/貫通修正の成否**(avatar × garment → 緑/直った/直らなかった) |
| 単位 | 1資産 = 1行 | 1組み合わせ = 1行 |
| 意味 | 体型地図のノード | **マッチング行列のセルの実測値** |

エッジ共有がこのプロジェクトでは決定的です。現在 `matching_matrix`([MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §7)は採寸照合で「この組は黄だろう」と**予測**しますが、その予測が当たったかの**正解ラベルがありません**。成否の持ち寄りは、その**予測層の正解データセット**になります。

つまり成否共有 ＝ **修正可能包絡 f([CORRECTABILITY_FIELD.md](CORRECTABILITY_FIELD.md))の経験的較正**。手元の n=1 の観測を、コミュニティ規模の証拠へ育てる道です。単なるデータ共有ではなく、f の等高線を実測で描く営みとして設計します。

## 2. 媒体 — スキーマ標準＋追記型 jsonl 公開台帳

中央 DB/API ではなく、**共有スキーマを標準として定義し、持ち寄りは追記型 jsonl の公開台帳(git 等)で行う**ことを本命に置きます。

```
[各ユーザー] ローカル採寸 ─→ 自分の保管庫(private; スカラーのみ)
                              │ 同意した行だけ export
                              ▼
[公開台帳] shared-bodies.jsonl / shared-fits.jsonl(追記型・寄稿は PR)
                              │ 誰でも clone・取り込み
                              ▼
[各ユーザー] 自分の保管庫に union ─→ 全体データで matching/cluster
```

### なぜ中央 DB を本命にしないか

- **No Cache の延長** — 本プロジェクトは「形状データを持たない」設計。中央 DB をホストすることは**データと運用責任と単一障害点を抱える**ことを意味する。非営利・オープン実装という目的に対して重く、自律性を損なう。
- **所有すべきはデータでなく形式** — 標準化するのは**スキーマ＋ハッシュ規約＋同一性解決**であって、データは各自/コミュニティが持つ。これは ECOSYSTEM_VISION の「診断を互換性標準へ育てる」の実体化。形式を媒体非依存に保てば、媒体は後から差し替えられる(§8)。

### 自己相似

生(寄稿しやすい・人が読める)＝**追記 jsonl**、配布(速い・小さい)＝**派生 sqlite/圧縮スナップショット**。これは保管庫レベルで採った「jsonl＝投入経路 / sqlite＝保管・問い合わせ」(MEASUREMENT_SPEC §5)が、コミュニティ規模でもそのまま同じ形で再帰します。ローカルとエコシステムが同じ二層なので、覚えることが増えません。

### 行スキーマ(検討用の叩き台)

```jsonc
// shared-bodies.jsonl — ノード
{ "kind":"body", "canonical":"unkt", "headCount":7.1, "radii":[...], "coverage":1.0,
  "meshHash":"841d90a0…", "conditions":{ "radiusPercentile":0.85, "scaleNorm":true, "shapekey":"0", "tool":"0.2.0" },
  "contributor":"anon-7af", "measuredAt":"2026-06-25T…" }

// shared-fits.jsonl — エッジ
{ "kind":"fit", "avatar":"unkt", "garment":"Manuka_1", "predicted":"YELLOW",
  "observed":"FIXED_OK", "method":"manual-bake", "residualMaxMm":3.0,
  "conditions":{…}, "contributor":"anon-7af", "measuredAt":"…" }
```

`observed` の語彙(FIXED_OK / FIXED_FAIL / NO_FIT_NEEDED / NOT_ATTEMPTED…)は §10 の未決。

## 3. 同定(identity) — 誰の採寸が同じ資産か

役割を厳密に分けます(混ぜると壊れる):

- **メッシュハッシュ = 版確認専用** — 同ハッシュ＋同条件 → 同採寸値が再現する「再現性キー」(MEASUREMENT_SPEC §6)。来歴・版固定・陳腐化検知に使う。
- **canonical 名 = 名寄せ** — 散らかったプレファブ名を正規名へ(同一性シソーラス、MEASUREMENT_SPEC §5)。
- **比較は採寸値で** — ユーザー間では import 設定/MOD で同じ資産でもハッシュが一致しないため、**ハッシュの完全一致を同定に使わない**。比較は採寸スカラーの近さで行う。

この「ハッシュは版、同定は値」の分離が、後段の誤り耐性・撤回の両方を支えます。

## 4. 誤り耐性 — 誤った結果が追記されたら

「誤り」は一様でないので分けます:

| 種類 | 例 | 機械で弾けるか |
|---|---|---|
| 測定ミス | 別素体で測った・正規化忘れ | 一部可(self-consistency) |
| 同定ミス | 別資産を unkt と命名 | 一部可(採寸値の外れ) |
| 環境差 | import/MOD で値がずれる(嘘ではない) | 不可(**比較ゲートで無害化**) |
| 悪意/いたずら | 直らないのに「直った」 | 巧妙なら不可(人間ゲート) |
| 陳腐化 | tool 版が古い・資産が改訂された | 可(version/hash で検知) |

### 鉄則: 削除ではなく supersede / retraction

追記台帳は append-only。誤行を物理削除すると ①履歴に残る ②他人が取り込み済み ③「なぜ消したか」が失われる。だから**後続の行で訂正**します:

- **訂正** — 同じ `(canonical, conditions)` の新しい `measuredAt` 行が古い行を supersede(最新版が勝つ)。版管理の仕組みがそのまま誤り訂正になる。
- **撤回** — `{ "kind":"retraction", "scope":"…", "reason":"…", "by":"…", "at":"…" }` を1行足し、クエリ側が除外。次のスナップショット(§6)から物理的に落とす。

消さないので**来歴が透明**に保たれます。

### 機械で弾ける層(self-consistency)

各行がハッシュ・条件・版を持つので矛盾が検出できます。特に重要なのは:

1. **同ハッシュなのに採寸値が割れる2行** → どちらか測定ミス(同形状なら同値が出るのが再現性キーの定義)。
2. **条件が違う行を比べさせない** → 正規化済・同 tool 系列の行だけ比較するゲート。環境差は消すのでなく**土俵を揃えて混ぜないことで無害化**。
3. **成否が予測と大きく食い違う** → 採寸上は緑なのに「直らなかった」が複数寄稿者から来る。

3つ目が二重の意味を持ちます: これは**誤り検出器であると同時に f の反例発見器**です。予測が外れた所こそ、f の等高線が想定とずれている場所。1人だけなら疑い(要再現)、複数から再現されたら**モデルの反例として昇格**。誤りを恐れて閉じるのでなく、**矛盾を可視化する**設計にすれば、悪意は弾かれ善意の反例は学びになります。

### 人間で弾く層

機械を抜けた分は **PR レビュー**で受けます。中央 API への自動投稿でなく寄稿を PR にすることで、明らかなスパム/嘘をマージ前に弾ける — **中央サーバを持たない設計が品質ゲートとして働く**。繰り返し矛盾する寄稿者の行はクエリ側で重みを下げる。中央権威でなく**来歴の透明性**で信頼を担保します。

### 正直な限界

条件まで整合させた**巧妙な悪意**は機械検出できません。最終的には PR レビュー＋評判＋「再現できる人が再現する」人間オラクルに依存します。成否は「真実」でなく**条件付きの観測**で、母数が増えれば分布で頑健化する性質のもの。台帳は裁定者でなく観測の集積です。

## 5. スケール — 同時大量書き込み・サイズ肥大

### まず規模感(過剰設計を避ける)

- 1行 ≈ 300〜500 バイト(**形状を持たないから小さい**)。10万行 ≈ 30〜50 MB。
- 実際に報告される (avatar×garment) は全組み合わせのごく一部。現実は数万〜数十万行。
- 書き込みは**有志の手作業の集積 = 1日数十〜数百行**のオーダー。

素のサイズも同時書き込み頻度も、当面は問題になりません。形状を捨てた設計がここでも効いています。

### 同時書き込みの真の敵は末尾コンフリクト

スループットでなく **git の構造**が敵です。1つの巨大 jsonl に全員が末尾追記すると**末尾追記同士が必ずコンフリクト**する。解決は分割:

- **シャーディング** — 寄稿者別/日付別/canonical 別にファイルを割る(`shared/fits/by-contributor/<id>.jsonl` 等)。各自は自分のファイルにだけ追記 ＝ 末尾コンフリクトが構造的に起きない。読む側は glob で union。
- 究極は **1報告 = 1小ファイル**(ファイル名 ＝ content hash)。絶対コンフリクトしないがファイル数が増える。
- **bot 集約**(issue/フォームで受け bot がシャードへ直列追記＋self-consistency 自動チェック)も選択肢。ただし bot ＝軽い運用主体で、自律性とのトレードオフ。

### 肥大の手当て(当面不要だが道筋)

1. **シャード** — 1ファイルを小さく保つ。clone は sparse-checkout で必要分だけ。
2. **log compaction** — supersede/retraction 済みの古い行を落としたスナップショットを定期生成(派生・再生成可)。生 log は履歴アーカイブへ。
3. **二形態配布** — git 本体は生 jsonl シャード(テキスト)、巨大な派生 sqlite は Release アセット/LFS。jsonl は gzip が10倍前後効く。

## 6. 撤回 — 製作者が後から公開を拒否したら

ここが最も難しく、**正直に限界から始めます**。分散追記型では「物理的な完全削除」を約束できません:

- **追記台帳は append-only な歴史を持つ**(行を消しても履歴に残る)
- **既に分散済み**(他人が取り込んだ派生からは回収不能)
- **No Cache とは別軸**(採寸スカラーは形状復元不可でも、「誰の数値を公開してよいか」は同意/ライセンス軸)

「忘れられる権利」の法的枠組みですら**「合理的な努力」**であって、第三者が既にコピーした分の完全回収は求めません(技術的に不可能だから)。同じ正直さで設計します — **約束できるのは「正典からの除去＋再配布の停止＋撤回の記録」まで**。これを最初から規約に書きます。

### 機構は誤り訂正と同じ

§4 の retraction をそのまま使います: `{ "kind":"retraction", "scope":"creator:X", "reason":"opt-out" }` 1行でクエリが当該製作者の全行を除外し、次のスナップショットから物理除去。シャーディングで製作者/寄稿者別にファイルが分かれていれば**そのファイルごと削除 = 1操作**。**分離設計(中央に混ぜない)が、撤回を局所操作に保ちます**。

### 本丸は事前同意(撤回を減らす)

- **オプトイン＋粒度明示** — 載せる前に「商品ページが身長を載せる粒度のスカラーのみ・形状なし・いつでも撤回可・ただし分散済みコピーは回収不能」を明示し同意。既定は載せない。
- **opt-out 窓口**(issue テンプレ/フォーム) — 中央サーバなしでも受付になる。
- **最強の防御 = 載せる粒度を最初から低く** — 回収不能でも痛くないデータ(商品ページ公開情報相当)しか出さない。No Cache が「形状を出さないことで守る」のと**同型**。

### 成否(不利な結果)の撤回は別質

「うちの衣装が貫通した」の撤回要請はプライバシーでなく**レピュテーション**問題です。集計度で解きます: 成否は「衣装の欠陥」でなく「(衣装×素体×条件) の観測」なので、個別名を伏せて**体型クラスタの傾向だけ残す**(例「この体型帯では厚物が残りやすい」)。個体保護と f の学習を両立。ただし小さい台帳では再同定が残る、と正直に置く。

### 最終手段

法的/強い要請があれば、履歴書き換えで当該行を除去(本リポジトリでも過去にコミットのメタデータを履歴から除去した実績あり)。ただし全 clone を無効化 ＝ 分散の前提が壊れるので**乱発しない最終手段**。

## 7. 撤回権の尊重は信頼を生む

撤回を尊重する仕組みを最初から持つこと自体が、本プロジェクトの「正直・権利尊重」の実証になります。No Cache が「形状を**持たない**ことで信頼を得る」のと同型で、撤回権の尊重は「**引っ込められること**で信頼を得る」。製作者が安心して載せられる ＝ 台帳が育つ、という正のループ。データの完全回収を約束しないからこそ、粒度を最初から低く絞り、引っ込められる設計にする — これが筋の通った一貫性です。

## 8. 媒体トレードオフ — 追記台帳(git)と Fossil/shun

撤回(§6)に対して、SCM 自体が消去機能を持つ選択肢があります。

**Fossil**(SQLite の作者による DVCS。リポジトリ全体が1つの SQLite ファイル、bug tracker/wiki/forum を内蔵、self-hosted)は **shun** という機能を持ちます — 特定の artifact をリポジトリから物理除去し、sync で再導入されないようにブロックする(誤投稿・違法コンテンツ除去のために設計)。一般的な「履歴書き換え＋強制 push」と違い、**消去が一級市民**である点が肝です。(shun リストの伝播挙動の細部は実装で要確認 — 本書の理解では、除去はローカルが基本で、設定共有による伝播も可能。)

### Fossil が刺さる点

1. **撤回が SCM ネイティブ** — §4-§6 でアプリ層に組んだ retraction/compaction を**インフラ層に委譲**できる。
2. **リポジトリ ＝ 1 SQLite が保管庫設計と物理一致** — MEASUREMENT_SPEC §5 で採った「sqlite を保管・問い合わせに」と同じ物理形式。カスタム採寸テーブルを同居させられる。
3. **思想が近い** — 単一ファイル・自己完結・依存最小・self-hosted は No Cache の自律性と整合的。中央ホスティングへの依存を減らせる。組み込み forum が opt-out 窓口・成否報告を自己完結させる。

### それでも残る限界(過信の罠)

shun は強力ですが**完全回収の不可能性は変わりません**:

- **ネットワーク外には及ばない** — export 済みの sqlite・スクリーンショット・無関心なフォークは管轄外。shun は回収範囲を協力的なネットワーク内に**広げる**だけ。
- **伝播は自発 sync 依存** — オフラインのクローンや敵対ノードは無視できる。
- **普及摩擦** — 既存の改変ワークフローは一般的な git ホスティングに馴染んでおり、寄稿ハードルが上がる(bot/web UI である程度隠せる)。

肝心なのは、**shun があっても §6 の「低粒度・オプトイン・回収不能の明示」は緩めない**こと。shun は「正典除去＋再配布停止」を**強化する道具**であって、「完全削除の約束」を可能にするものではない。「消えるから安心」と過信して粒度を上げるのが一番危ない罠です。

### 媒体非依存の原理

「git か Fossil か」は二択ではありません。**所有するのはスキーマ(場)であって媒体ではない**ので:

- **撤回を最重視するなら Fossil**(shun ＋ SQLite 一致 ＋ 自律性)
- **普及を最重視するなら git ホスティング＋bot**(既存ワークフローの網の中)
- **どちらでもスキーマは同じ** ＝ 規模と普及を見て後から乗り換えられる

採寸スキーマ・ハッシュ規約・同一性解決・retraction セマンティクスを媒体非依存に保つことが、この柔軟性の前提です。

## 9. 普及の階段(共有レイヤ版)

ECOSYSTEM_VISION §8 と同じく「一人でも便利」を積みます:

1. **ローカル保管庫** — 自分の採寸コーパスを sqlite で持つ(landed の延長)。共有なしで価値が完結。
2. **二者間の手渡し** — jsonl を1ファイル渡すだけで相手の地図に自分のノードが増える。プロトコル不要。
3. **小さな公開台帳** — 有志のシャード追記。self-consistency と PR レビューで品質を保つ。
4. **f の経験的較正** — エッジ(成否)が貯まり、予測 vs 実測でマッチング行列を検証。等高線が描ける。

各段は前段の自然な延長で、どこで止まってもそこまでの価値が残ります。

## 10. 未決事項(検討の宿題)

- **スキーマの確定** — `body`/`fit`/`retraction` 各行の正式フィールド。ECOSYSTEM_VISION の FIT_REPORT_FORMAT(分離予定)と整合させるか統合するか。
- **`observed` の語彙** — FIXED_OK / FIXED_FAIL / NO_FIT_NEEDED / NOT_ATTEMPTED / FORCE_APPLIED… の定義と、residual との関係。
- **寄稿者 ID の匿名性** — 一貫した匿名 ID(評判の蓄積)とプライバシーの両立。
- **同意取得の具体プロセス** — 製作者への問い合わせテンプレ、粒度の規約文、opt-out の SLA。
- **成否撤回の集計度** — レピュテーション保護の k-匿名性しきい値(何件集まれば個別名を伏せて傾向化するか)。
- **媒体の最終選択** — git ホスティング / Fossil / その他。規模と普及の実データを見て保留。今はスキーマを媒体非依存に保つことだけ確定。
- **基準マネキン**(ECOSYSTEM_VISION §9 と共通) — 再配布自由な数体を共通の物差しに。検証の足場。
- **compaction の運用** — スナップショット生成の頻度・トリガ、生 log アーカイブの置き場。
