# 関節部の可変マージン — 設計深掘りと判断整理

このドキュメントは**検討資料**です。ROADMAP フェーズ3「関節部の可変マージン」の実装前深掘りとして、何が必要か・どこに設計判断があるか・隣接する「本命(代表ポーズ対応)」とどう競合するかを整理します。背景は [DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) §6、現行パイプラインは [DESIGN.md](DESIGN.md) §6、開発計画は [ROADMAP.md](../ROADMAP.md) フェーズ3。最終更新: 2026-06-18(下調べ段階。着手判断は本書 §8 の通り保留)。

## 0. 一行まとめ

可変マージンは「いま貫通していない関節付近の布を、後の実行時の曲げに備えて予め体から離す」**予防策**で、現行の「侵入がある所しか触らない」設計からの逸脱になる。実装は加算的に低リスクで書けるが、(A)検出に効かせるか押し出しだけか、という真の設計判断が一つあり、さらに **posed capture が既に動くため「本命=代表ポーズ対応」へ一段飛ばす方が ROI が高い可能性**がある。判断は保留(§8)。

## 1. 問題の正体 ―「予防的マージン」という特異点

[DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) §6 によれば、関節貫通は **素体と衣装のウェイト塗りの差 × 関節回転角**で生じ、**体型差がゼロでも起きる**。ここに本機能特有の難しさがある:

- 本ツールはエディタ時にキャプチャした**静止ポーズ(通常 T/A ポーズ)**で動く。静止ポーズでは曲げが無く、**関節貫通はそもそも検出されない**。
- したがって可変マージンは、検出した貫通を直す現行の全機能と異なり、**「いま貫通していない頂点を、後の実行時の曲げに備えて予め体から離す」**もの。これは「検出ベースなので侵入がある所しか触らない」([DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) §6)という設計思想からの逸脱になる。

この「静止時に余裕(standoff)を作る予防策」という性質が、以下すべての判断の根になる。

## 2. 現状の配線 ― margin はスカラーで一本貫通している

`margin`(float)が次を一本で貫く:

| 箇所 | 使い方 |
|---|---|
| `PenetrationDetection.Scan` | `distance < margin` で hit 判定 |
| `PenetrationPushOut.Apply` | `collider.PushOut(point, margin)` で margin 上面へ押す |
| `PenetrationSolver.Solve` / `SolveProjected` | Scan/PushOut を**複数回**呼ぶ。最終チェックは `margin - 1e-4` |
| `PreflightDiagnostic.Evaluate` | `surfaceDepth = hit.depth - margin`(深さの基準面) |

頂点別化は **加算的なシグネチャ(`float[] perVertexMargin = null`、既定 null=現行挙動)**で書けば既存テストへの影響はゼロにできる。ただし上記4ファイルの全署名に波及する。`CollapsedVertices` のような単独追加では済まない点が、本項目を「最大の変更」にしている。

## 3. 設計フォーク①(最重要)― 検出に効かせるか、押し出しだけか

深掘りで出てきた本質的な分岐。

### A. 押し出しのみ拡大

検出は base margin のまま(新規 hit を作らない)。関節跨ぎの**既に貫通している**頂点だけ、押し出し先を 1.5〜2× へ伸ばす。

- **利点**: 安全・最小。静止ポーズで余計な膨らみが出ない。preflight も無影響。
- **欠点**: 静止時に貫通していない袖口・裾には**何も起きない** → 予防効果がほぼ無い。§6 の意図(曲げの前に余裕を作る)を満たさない。

### B. 検出も拡大

関節跨ぎ頂点は inflated margin 内にあれば hit にし、inflated margin 上面へ押す。

- **利点**: 静止時に貫通していない布も体から離れる=本来欲しい**予防的 standoff**。
- **欠点**: 過剰だと静止ポーズで肘/膝に隙間が出る(実行時クリップを静止時ギャップに置換しただけになる)。hit セマンティクスが変わり、preflight の扱い(§6)に注意が要る。

§6 の「曲げ起因の貫通への保険」を素直に読むと **B が意図**だが、B は実装量もリスクも大きい方。**ここは暫定デフォルトで流せない真の設計判断**であり、本項目を保留にする主因の一つ。

## 4. 関節跨ぎ頂点の定義(提案デフォルト ― 判断不要で固められる)

入力は既存データで足りる: `SkinnedMeshRenderer.sharedMesh.boneWeights`(頂点あたり最大4ボーン)と `renderer.bones`(`Transform[]`)。逆スキニングで既に同種データを扱っている([SkinningMath](../Assets/VRCloth-Declipper/Core/SkinningMath.cs))。

> **頂点が関節を跨ぐ** ⇔ 有意ウェイト(例: > 0.1)が、スケルトン上で**祖先‐子孫関係にある異なる2ボーン**に乗っている。

親子関係にある2ボーンは互いに回転しうる=その間に関節がある。UpperArm+LowerArm=肘、UpperLeg+LowerLeg=膝、Neck+Chest=襟、LowerLeg+Foot=ソックス上端 ― §6 が挙げる**肌見せ境界(袖口⇔肘、裾⇔膝)そのもの**を自然に拾う。

[CollapsedVertices](../Assets/VRCloth-Declipper/Core/CollapsedVertices.cs) と同様、純 Core の分類器(boneWeights + 親子マップ → 跨ぎ頂点集合)としてユニットテスト可能。親子マップは `renderer.bones[i].parent` の連鎖から構築できる。**ここはデフォルトで決められ、判断は不要**。

## 5. 増幅率と較正

§6 が **1.5〜2×** を明示。暫定定数として置けるが(他のしきい値と同じ扱い)、**「効くか・隙間が出ないか」の確認は曲げポーズでの目視が必須**で、[E2E_TEST_GUIDE.md](E2E_TEST_GUIDE.md) 系の pose-gated 較正になる。機構は今作れるが**較正値の確定は目視待ち**。

## 6. Preflight は base margin を維持(推奨で固められる)

inflated margin を verdict 計算に混ぜると、関節頂点の貫通が「浅く」見えて**実貫通を取りこぼす**方向(最悪方向)に振れる。可変マージンは**ソルバの押し出し先**にだけ効かせ、RED/GREEN 判定は base margin で取るのが正しい。これは判断というより設計上の帰結で、先に固定できる。

## 7. 戦略的論点 ― 可変マージン vs 代表ポーズ対応

[DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) §6 の line「任意ポーズでの修正は現状でも可能」が効いてくる: キャプチャ(`BakeMesh`)も書き戻し(逆スキニング)も**ポーズ対応**なので、**エディタで関節を曲げてから Run すれば、そのポーズで直せる**(`SkinnedRoundTripTests` が整合を保証)。そして §6 は可変マージンを**「安価な中間策」、本命は「代表ポーズ対応」(複数ポーズで解いて結果を束ねる)**と位置づけている。

| | 可変マージン(#4) | 代表ポーズ対応(本命) |
|---|---|---|
| 実装 | per-vertex margin 配線(4ファイル)+ 跨ぎ分類器 + フォーク①の判断 | 既存 posed capture を N ポーズ回す + 変位の合成(頂点別 max push 等) |
| 効果 | 粗い予防 standoff(当て推量) | 実際の曲げでの貫通を直接解く |
| 較正 | 増幅率を目視較正 | ポーズ選定を目視 |
| §6 の位置づけ | 代用 | 本命 |

**posed capture が既に動く以上、本命の追加コストは「ポーズ駆動 + 変位合成」だけ**で、per-vertex margin 配線が要らない。つまり **#4 に投資するより本命へ一段飛ばす方が ROI が高い可能性**がある。少なくとも「#4 は本命の劣化版で、配線コストは本命と大差ない」ことは深掘りで確定した。

> 補足: 代表ポーズ対応も独自の未決(どのポーズ集合を標準とするか、レンダラー別 max push の合成で破綻が出ないか、ポーズ駆動を Humanoid muscle で行うか直接 Transform 回転か)を持つ。本書はあくまで #4 の深掘りであり、本命の本格設計は別途。

## 8. 後回し中に決める判断リスト

実装着手前に確定したいもの:

1. **フォーク①(§3)**: 検出に効かせる(B・予防的)か、押し出しのみ(A・安全)か。#4 をやるなら最初の決め。
2. **そもそも #4 をやるか、本命(代表ポーズ対応)へ飛ばすか(§7)**。現時点の私見は **本命を先に小さく試作する方が筋が良い**(posed capture 流用、per-vertex margin 不要、§6 の本命)。
3. **増幅率(1.5〜2×)とポーズ選定**は、いずれにせよ目視較正待ち(§5)。

判断不要で先に置けるもの(着手が決まれば即書ける):

- 関節跨ぎ分類器(§4)
- Preflight = base margin 維持(§6)
- per-vertex margin の加算的シグネチャ(§2、既定 null=現行挙動)

---

*可変マージンは「曲げの前に余裕を作る」予防策であり、検出ベースの現行設計の外にある。実装の壁はスカラー margin の頂点別化(加算的で低リスク)、設計の壁はフォーク①と「本命へ飛ばすか」。判断が固まるまで保留。*
